## Правила для Claude Code

1. **Не вноси изменения без явного согласования** — меняй только то, о чём просил пользователь. Заметил другую проблему в соседнем файле — сообщи отдельно: «Вижу ещё одну проблему — [описание]. Исправить?» Не трогай молча.
2. **Один промпт = одна логическая единица** — не добавляй «бонусные» улучшения, даже очевидно полезные.
3. **Plan before code** — сначала письменный план (что делаем, зачем, как, что не трогаем), пользователь утверждает или корректирует, только потом реализация.
4. **Верификация файлов — блокирующее правило** — никакого промпта без прочтения полного актуального содержимого каждого затрагиваемого файла. Саммари вида «этот файл содержит...» не принимаются. Без исключений.
5. **Регрессионный щит обязателен** в каждом промпте, который меняет существующий код: явный список того, что нельзя трогать — файлы, функции, поведение. Claude Code склонен «улучшать» соседний код — это нужно явно блокировать.
6. **Все тексты на русском, код и @rule-комментарии на английском.**
7. **Git-коммит после каждого завершённого шага**, не в конце всей задачи.
8. **Не скачивай и не распознавай повторно** то, что уже есть локально — если файл уже на VDS, работай с локальной копией.
9. **Если ошибка не решается со второго раза** — предложи минимум 3 принципиально разных варианта решения. Пользователь выбирает — Claude Code действует.
10. **Каждый новый файл** начинается с шапки: `@file`, `@description`, `@dependencies`, `@rule`, `@lastModified`.
11. **Каждая экспортируемая функция** имеет JSDoc с `@input` / `@output`.
12. **Файлы >100 строк** — с регион-комментариями (`// ─── SECTION ───────`).
13. **Файлы >200–300 строк** — сигнал к декомпозиции. Один файл = одна ответственность. Меньший размер файлов снижает риск регрессий при правках Claude Code.
14. **Мёртвый код — удалять**, не комментировать.
15. **При любых изменениях в коде — обновлять соответствующие файлы `knowledge/`**. Это не опциональный шаг — это часть Acceptance Criteria каждого промпта. Если изменился интерфейс, архитектура, инфраструктура, правило или задача в roadmap — knowledge-файл должен отражать новое состояние.
16. **Не сокращать формулировки** при переносе или пересохранении инструкций, ТЗ, контекста, правил — переносить в исходном виде. Новый текст писать лаконично, но без потери смысла и без возможности разного толкования.
